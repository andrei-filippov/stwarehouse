## Пошаговый план

### Фаза 1: База данных (SQL)
**Файл:** `supabase_link_equipment.sql`

1. Добавить `inventory_id` в таблицу `equipment`
2. Добавить `equipment_id` в таблицу `cable_inventory`
3. Создать индексы
4. Мигрировать существующие данные (по совпадению имени)
5. Создать триггер для синхронизации quantity
6. Создать функцию `create_equipment_linked()` для создания в обеих таблицах

### Фаза 2: Frontend - Создание оборудования
**Файлы:** 
- `src/hooks/useEquipment.ts` - обновить `addEquipment()`
- `src/components/EstimateBuilder.tsx` - без изменений (работает через хук)

**Что меняем:**
```typescript
// useEquipment.ts - addEquipment()
// Было: создаём только в equipment
// Стало: создаём в equipment + cable_inventory + связываем
```

### Фаза 3: Frontend - QR сканер
**Файл:** `src/components/QRScanPage.tsx`

**Что меняем:**
```typescript
// Запрос резервов из смет
// Было: ищем по equipment_id = item.id (не работает!)
// Стало: ищем по equipment.inventory_id = item.id

const { data: estimateReservations } = await supabase
  .from('equipment')
  .select('estimate_items(quantity), estimates!inner(event_date)')
  .eq('inventory_id', item.id)  // Новая связь!
  .gte('estimates.event_date', today);
```

### Фаза 4: Синхронизация количества
**Триггер в SQL:**
- При изменении `cable_inventory.quantity` → обновляем `equipment.quantity`
- Сметы всегда видят актуальное количество

## Преимущества этого подхода

✅ **Обратная совместимость**
- Все существующие сметы продолжают работать
- Ничего не сломается у пользователей

✅ **Постепенная миграция**
- Старые записи связываем по имени (одноразово)
- Новые создаём сразу связанными

✅ **Конструктор смет не меняется**
- Он продолжает работать с `equipment`
- Но теперь `equipment` знает о складе

✅ **QR сканер получает резервы**
- Может найти оборудование через связь
- Показывать реальное "Свободно"

## Риски и митигация

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Дублирование оборудования | Средняя | Проверять по имени перед созданием |
| Рассинхронизация quantity | Низкая | Триггер в БД + периодическая проверка |
| Пустые inventory_id у старых | Высокая | Запустить скрипт миграции |

## Тестирование

1. Создать оборудование в смете → проверить появление в складе
2. Отсканировать оборудование → проверить видимость резервов
3. Изменить количество на складе → проверить обновление в сметах
4. Создать смету с резервом → проверить "Свободно" в QR сканере
