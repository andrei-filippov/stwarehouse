# Анализ готовности проекта к SaaS

**Дата:** 2026-04-17  
**Проект:** СкладОборуд (stwarehouse)  
**Стек:** React 19 + Vite 7 + TypeScript + Tailwind + Supabase  
**Деплой:** Vercel

---

## Общая оценка: 4.5 / 10

Проект представляет собой **функциональное B2B-приложение с элементами мультитенантности**, но до полноценного SaaS-решения не дотягивает по ключевым направлениям: **монетизация, операторский контроль, безопасность и DevOps**.

---

## 1. Архитектура и мультитенантность

**Оценка: 7 / 10**

### Что реализовано ✅
- **Таблица `companies`** со `slug` (для поддоменов/путей), `plan`, `plan_expires_at`
- **`company_id`** добавлен во все бизнес-таблицы через миграции
- **`company_members`** — связь user↔company↔role с статусами (`pending`/`active`/`suspended`/`removed`)
- **Переключение между компаниями** (`switchCompany`) — пользователь может состоять в нескольких
- **Поддоменная поддержка** (`getSubdomain()`, `getSlugFromPath()`, `/c/:slug`)
- **RLS-политики** основаны на проверке членства в компании через `company_members`
- **SECURITY DEFINER функции** для создания компании и управления членами (`create_company_with_owner`, `invite_company_member`)

### Проблемы ⚠️
- **131 SQL-файл** в корне проекта — хаотичная история миграций, нет порядка применения
- **Нет версионирования миграций** (не используется Supabase CLI migrations)
- **Нет soft deletes** — данные удаляются физически, невозможно восстановить
- **Нет tenant isolation на уровне Postgres schema** — все тенанты в одной схеме `public`

### Рекомендации
1. Перейти на `supabase db push` / CLI-миграции
2. Добавить `deleted_at` + `deleted_by` на все бизнес-таблицы
3. Рассмотреть разделение по schema для enterprise-клиентов

## 2. Аутентификация и авторизация

**Оценка: 6 / 10**

### Что реализовано ✅
- **Supabase Auth** — email/password, сессии, авто-рефреш токена
- **Профили** создаются автоматически через DB trigger
- **6 ролей:** owner, admin, manager, warehouse, accountant, viewer
- **Кастомные разрешения** — таблица `user_permissions` + RPC `get_user_effective_permissions`
- **Приглашения** через RPC `invite_company_member` + Edge Function для email
- **Offline fallback** — кэширование пользователя в IndexedDB

### Критические проблемы ❌
- **Доступ к табам проверяется ТОЛЬКО на клиенте** (`hasAccess()` в `permissions.ts`). Любой пользователь может вызвать Supabase API напрямую и получить/изменить данные, если RLS политика не строгая
- **Нет MFA/2FA**
- **Нет SSO** (Google Workspace, SAML, OIDC)
- **Нет rate limiting на попытки входа**

### Рекомендации
1. Добавить server-side middleware для проверки доступа к табам (RPC или Edge Functions)
2. Включить MFA через Supabase
3. Добавить rate limiting на auth endpoints

## 3. База данных (Supabase)

**Оценка: 7 / 10**

### Что реализовано ✅
- **RLS включён** на большинстве таблиц
- **Политики** корректно фильтруют по `company_id` через проверку `company_members`
- **Audit logs** — таблица `audit_logs` + триггеры на 6+ таблиц (checklists, equipment, cable_inventory, и т.д.)
- **Индексы** на `company_id`, `user_id`, даты

- **QR-коды** с авто-генерацией `generate_qr_code()`

### Проблемы ⚠️
- **Хаотичные миграции** — 131 файл без нумерации, сложно понять порядок
- **Часть таблиц может не иметь RLS** — нужен аудит всех таблиц
- **Нет automated backups strategy** в коде/документации

## 4. Платежи и монетизация (КРИТИЧЕСКИ ВАЖНО)

**Оценка: 2 / 10 — БЛОКЕР ДЛЯ SaaS**

### Что реализовано ✅
- Поле `plan` в `companies` (`free`/`basic`/`pro`/`enterprise`)
- Поле `plan_expires_at`
- RPC `create_company_with_owner` принимает `p_plan`

### Что ОТСУТСТВУЕТ ❌❌❌
- **НЕТ платёжной интеграции** — ни Stripe, ни ЮKassa, ни CloudPayments
- **НЕТ лимитов/квот по планам** — нет проверок на количество пользователей, смет, оборудования
- **НЕТ billing portal** — пользователь не может посмотреть тариф, обновить план, ввести карту
- **НЕТ trial-логики** — нет пробного периода, автоматических напоминаний
- **НЕТ invoices** — клиенты не получают счета
- **НЕТ webhooks** для обработки платежей
- **НЕТ grace period** при окончании подписки

### Рекомендации
1. Интегрировать **ЮKassa** (для РФ) или **Stripe** (для международного рынка)
2. Создать таблицу `subscriptions` с `status`, `current_period_start/end`
3. Добавить лимиты: `max_members`, `max_estimates`, `max_equipment` по планам
4. Создать страницу `/billing` с выбором плана
5. Добавить Edge Function для обработки webhook'ов платежей

## 6. Безопасность

**Оценка: 5 / 10**

### Что реализовано ✅
- **CSP** в `index.html`
- **RLS** политики на таблицах
- **Audit logs** с IP и user_agent
- **Анонимный ключ Supabase** используется корректно (RLS защищает)

### Проблемы ⚠️
- **CSP слишком мягкий:**
  - `connect-src *` — разрешает любые исходящие соединения
  - `script-src 'self' 'unsafe-inline'` — разрешает inline-скрипты (XSS-риск)
  - `frame-src *` — разрешает любые iframe
- **Нет rate limiting** на API
- **Нет WAF/DDoS защиты** (кроме того, что даёт Vercel/Cloudflare)
- **Нет CSP nonce / hash** для inline-скриптов
- **Anon key встроен в бандл** — нормально при RLS, но лучше ротировать

### Рекомендации
1. Ужесточить CSP:
   ```
   connect-src 'self' https://*.supabase.co https://*.yandex.ru ...;
   script-src 'self';
   frame-src 'none';
   ```
2. Добавить rate limiting через Supabase Edge Functions или API Gateway
3. Настроить Cloudflare или Vercel Firewall Rules
4. Добавить `Strict-Transport-Security`, `X-Frame-Options`, `X-Content-Type-Options`

## Вывод

Проект **не готов к запуску как SaaS** без минимум **4-6 недель** доработок критичных компонентов:

1. **Платёжная интеграция** — без этого нет бизнес-модели
2. **SaaS-оператор панель** — без этого невозможно управлять клиентами
3. **Лимиты по планам** — иначе бесплатные пользователи будут использовать бесконечно
4. **Error tracking + monitoring** — иначе вы будете слепыми к проблемам
5. **CI/CD** — ручной деплой на продакшн неприемлем для SaaS

Фундамент (мультитенантность, RLS, RBAC, PWA) **заложен хорошо**, но верхние слои SaaS (монетизация, операторский контроль, compliance) **практически отсутствуют**.
